# Workspice v1 仕様書

## 1. 文書の目的
本書は、Windows 上の作業環境をプロファイル単位で切り替える常駐アプリケーション「Workspice」の v1 実装仕様を定義する。

`docs/MEMO.md` に記載された構想を、実装・テスト・タスク分解に直接利用できる粒度へ整理し、以下を確定する。

- 機能要件
- 非機能要件
- データモデル
- UI/操作フロー
- アーキテクチャ
- 状態遷移
- 異常系の扱い
- テスト観点

## 2. 製品概要
Workspice は Windows のタスクトレイに常駐し、作業対象プロジェクトごとに定義した「プロファイル」をワンクリックで切り替えるデスクトップアプリケーションである。

プロファイル切替により、対象プロファイルに紐づく以下の構成を適用する。

- 起動時アクションの実行
- 現在プロファイルの終了時アクションの実行
- 壁紙の切替
- アプリ内部状態の更新
- 実行ログの記録

本アプリは単一ユーザのローカル利用を前提とし、v1 では外部 API やネットワーク通信に依存しない。

## 3. スコープ
### 3.1 対象
- Windows 10 / Windows 11 のデスクトップ環境
- 単一ユーザ利用
- ローカル完結の設定管理
- タスクトレイ常駐アプリ
- C# / .NET 8 / WPF による実装

### 3.2 v1 の対象機能
- タスクトレイからのプロファイル切替
- プロファイルの新規作成、編集、削除
- 起動時アクションと終了時アクションの定義
- アプリケーション起動
- コマンド実行
- Windows サービス開始/停止
- 事前条件チェック
- 事後確認
- 切替全体確認ダイアログ
- アクション個別確認ダイアログ
- 壁紙の切替
- 壁紙自動生成
- 実行ログの保存
- 前回アクティブプロファイルの記憶

### 3.3 v1 の対象外
- クラウド同期
- ネットワーク API
- プロファイル共有
- インポート/エクスポート
- マルチユーザ共有
- リモート操作
- 自動ロールバック
- テーマ切替
- プラグイン配布

## 4. 前提・制約
- アプリ本体は管理者権限で起動する前提とする。
- 各アクションは `Standard` または `Elevated` の実行権限を個別設定できる。
- 起動時に前回アクティブだったプロファイル情報は復元するが、切替処理の自動再実行はしない。
- 切替実行中に別の切替要求が来た場合は、保留せず即時拒否する。
- オフラインで完結することを必須要件とする。

## 5. 用語定義
- プロファイル: 作業環境のまとまりを表す設定単位。
- 起動時アクション: プロファイル適用時に実行する処理列。
- 終了時アクション: 現在プロファイルを離脱する際に実行する処理列。
- 事前条件チェック: アクション実行前に「実行が必要か」を判定するチェック。
- 事後確認: アクション実行後に期待状態へ到達したかを判定する確認処理。
- 安定状態: 切替処理中ではなく、アプリが一貫した現行プロファイルを保持している状態。
- AttentionRequired: 起動時アクション中の失敗により、ユーザ判断を要する中間状態。

## 6. 想定ユースケース
### 6.1 プロファイルを切り替える
ユーザはタスクトレイアイコンをクリックし、一覧から対象プロファイルを選択する。アプリは切替確認ダイアログを表示し、確認後に終了時アクション、起動時アクション、壁紙変更、状態更新、ログ出力を行う。

### 6.2 プロファイルを編集する
ユーザはタスクトレイメニューから編集画面を開き、プロファイル名、壁紙、アクション群、条件チェックなどを追加・編集・削除する。

### 6.3 起動済み判定により不要な起動を避ける
ユーザはアクションに事前条件チェックを設定し、既に対象アプリが起動済みである場合はそのアクションを自動的にスキップする。

### 6.4 実行後の状態確認を行う
ユーザはアクションに事後確認を設定し、プロセス起動やサービス状態の到達を確認できる。

## 7. 機能要件
### 7.1 タスクトレイ常駐
- アプリは起動後にタスクトレイへ常駐する。
- メインウィンドウ常時表示は行わない。
- タスクトレイメニューには以下を表示する。
  - 現在の状態
  - プロファイル一覧
  - プロファイル編集
  - 終了

### 7.2 プロファイル管理
プロファイルは以下の項目を持つ。

- `Id`
- `Name`
- `Description`
- `WallpaperMode`
- `WallpaperPath`
- `LaunchActions`
- `ShutdownActions`

要件:

- プロファイルの新規作成、編集、削除ができること。
- プロファイル名は必須入力とする。
- プロファイル ID はアプリ内部で採番する一意識別子とする。
- 編集 UI では起動時アクションと終了時アクションを分離表示する。
- 起動順序は一覧の並び順で定義する。

### 7.3 プロファイル切替
- 切替前に必ず全体確認ダイアログを表示する。
- 切替順序は以下で固定する。
  1. 現在プロファイルの終了時アクションを順に実行
  2. 切替先プロファイルの起動時アクションを順に実行
  3. 壁紙を切替先設定へ変更
  4. アクティブプロファイル状態を更新
  5. 実行結果をログ保存
- 現在プロファイルが未設定の場合、終了時アクションは実行しない。
- 切替先と現在プロファイルが同一の場合も確認ダイアログを表示し、ユーザが承認した場合のみ再適用を行う。

### 7.4 アクション定義
各アクションは共通基底 `ActionDefinition` と型別詳細から構成する。

共通項目:

- `Id`
- `Name`
- `ActionType`
- `TimeoutSec`
- `PromptBeforeRun`
- `Enabled`
- `Preconditions`
- `PostCheck`

型別詳細:

- `ApplicationLaunchAction`
  - `ExecutablePath`
  - `Arguments`
  - `WorkingDirectory`
  - `RunLevel`
- `CommandExecutionAction`
  - `FileName`
  - `Arguments`
  - `WorkingDirectory`
  - `RunLevel`
- `WindowsServiceControlAction`
  - `ServiceName`
  - `Operation`

共通要件:

- 無効化されたアクションは実行しない。
- `PromptBeforeRun = true` の場合、アクション実行直前に個別確認ダイアログを表示する。
- 個別確認で拒否された場合、そのアクションは `Skipped` として記録する。
- タイムアウト時は失敗扱いとする。

### 7.5 サポートするアクション種別
#### 7.5.1 ApplicationLaunch
- 指定 executable を起動する。
- 必要に応じて引数、作業ディレクトリを渡す。
- `RunLevel` を保持する。
- v1 ではアプリ本体が管理者権限で常駐する前提のため、`Standard` と `Elevated` の実行差は未実装とし、将来互換のため設定だけ保持する。

#### 7.5.2 CommandExecution
- コマンドラインを実行する。
- 標準出力、標準エラー、終了コードを取得できること。
- コマンドは `FileName` と `Arguments` で表現し、`cmd.exe /c` または PowerShell などの明示的なランナー指定を許容する。

#### 7.5.3 WindowsServiceControl
- 対象 Windows サービスに対して `Start` または `Stop` を実行する。
- 対象はサービス名で識別する。
- 実行後は指定がある場合のみ事後確認を行う。

### 7.6 事前条件チェック
事前条件チェックは、アクション実行前にコマンドベースで評価する。

`CheckDefinition`:

- `Command`
- `Arguments`
- `SuccessExitCodes`
- `OutputRegex`
- `Negate`

要件:

- チェックは複数設定可能とする。
- すべてのチェックが成功した場合のみアクションを実行する。
- いずれかのチェックが失敗した場合、アクションは `Skipped` として扱う。
- チェック判定には終了コードと出力正規表現の両方を利用可能とする。
- `Negate = true` の場合、判定結果を反転する。

### 7.7 事後確認
事後確認はアクション実行後に期待状態を確認する。

`PostCheckDefinition` は共通項目と型別詳細から構成する。

共通項目:

- `TimeoutSec`
- `PollIntervalMs`

型別詳細:

- `ProcessExistsPostCheck`
  - `ProcessName`
- `CommandCheckPostCheck`
  - `Check`
- `ServiceStatePostCheck`
  - `ServiceName`
  - `ExpectedState`

要件:

- 設定されたタイムアウト内で条件成立を監視する。
- 監視はポーリング方式とする。
- 監視失敗時はアクション失敗とする。
- `CommandCheck` モードでは内部的に `CheckDefinition` 相当の判定を再利用できる構造にする。

### 7.8 壁紙切替
- プロファイル適用時に壁紙を変更する。
- `WallpaperMode` は以下を持つ。
  - `CustomImage`
  - `GeneratedFromProfileName`
- `WallpaperPath` が設定されている場合はその画像を使用する。
- 未設定の場合はプロファイル名から画像を自動生成する。
- 生成画像はキャッシュして再利用する。

### 7.9 アプリ起動時の復元
- 前回アクティブだったプロファイル ID を読み込む。
- 画面表示やメニュー上では前回状態を示す。
- 起動時に自動でプロファイル切替処理は行わない。

### 7.10 実行ログ
- すべての切替実行についてログを残す。
- ログには少なくとも以下を含める。
  - 開始時刻
  - 終了時刻
  - 切替元プロファイル
  - 切替先プロファイル
  - 各アクションの実行結果
  - 失敗理由
  - ユーザ選択内容

## 8. 失敗時の扱い
### 8.1 基本方針
- アクション失敗時はその場で切替を停止する。
- ユーザに以下の選択肢を提示する。
  - 再試行
  - 残りを続行
  - 中止

### 8.2 停止時点による状態
- 終了時アクション中に中止された場合、最後に安定していたプロファイルを現行扱いのままとする。
- 起動時アクション開始後に中止された場合、状態を `AttentionRequired` とする。
- `AttentionRequired` 中はタスクトレイ表示に警告を含める。
- アクション失敗後にユーザが `残りを続行` を選び、切替シーケンス全体が完了した場合、最終状態は `ActiveProfile(targetProfileId)` とする。
- `残りを続行` により継続した失敗は警告付きで実行ログへ残す。

### 8.3 ロールバック
- v1 では自動ロールバックは実装しない。
- 復旧は再試行または別プロファイルへの手動切替で行う。

## 9. 状態遷移
アプリ内部状態 `AppState` は以下を持つ。

- `NoActiveProfile`
- `ActiveProfile(profileId)`
- `Transitioning(fromProfileId, toProfileId)`
- `AttentionRequired(lastStableProfileId, failedTargetProfileId)`

遷移ルール:

1. 初回起動時は `NoActiveProfile`。
2. 安定状態で切替開始すると `Transitioning`。
3. 切替成功時は `ActiveProfile(targetProfileId)`。
4. 終了時アクション中の失敗で中止した場合は `ActiveProfile(lastStableProfileId)` に戻る。
5. 起動時アクション中の失敗で中止した場合は `AttentionRequired(lastStableProfileId, targetProfileId)`。
6. `AttentionRequired` から再試行または別プロファイル切替に成功した場合は `ActiveProfile(...)`。

## 10. UI 要件
### 10.1 タスクトレイメニュー
- 先頭に現在状態を表示する。
- プロファイル一覧ではアクティブなプロファイルを視覚的に区別する。
- `AttentionRequired` 時は警告表示を追加する。
- 実行中は切替メニューを無効化する。

### 10.2 切替確認ダイアログ
- 切替元と切替先を表示する。
- 実行される終了時アクションと起動時アクションの件数を表示する。
- `実行` と `キャンセル` を持つ。

### 10.3 アクション確認ダイアログ
- アクション名、種別、対象、引数を表示する。
- `実行` `スキップ` `中止` を選択できる。

### 10.4 プロファイル編集ダイアログ
- プロファイル一覧と詳細編集領域を持つ。
- 起動時アクションと終了時アクションは別タブまたは別セクションで扱う。
- 壁紙はファイル指定または自動生成を選択できる。

### 10.5 アクション編集ダイアログ
- アクション種別ごとに入力項目を切り替える。
- 事前条件チェックは複数行で追加・削除できる。
- 事後確認はモード別の入力 UI を持つ。

## 11. データ保存仕様
### 11.1 保存先
- 設定ファイル: `%AppData%\Workspice\profiles.json`
- 実行ログ: `%LocalAppData%\Workspice\logs\`
- 生成壁紙キャッシュ: `%LocalAppData%\Workspice\wallpapers\`

### 11.2 設定モデル
`AppSettings` は以下を持つ。

- `SchemaVersion`
- `Profiles`
- `LastActiveProfileId`
- `StartWithWindows`
- `LogRetentionDays`

`StartWithWindows` が有効な場合、ログオン時に最高権限でアプリを起動する Windows タスクスケジューラ登録を行う。

### 11.3 JSON 例
```json
{
  "schemaVersion": 1,
  "lastActiveProfileId": "profile-dev-a",
  "startWithWindows": false,
  "logRetentionDays": 30,
  "profiles": [
    {
      "id": "profile-dev-a",
      "name": "開発A",
      "description": "WSL + Docker + VSCode 構成",
      "wallpaperMode": "CustomImage",
      "wallpaperPath": "C:\\Users\\user\\Pictures\\dev-a.png",
      "launchActions": [],
      "shutdownActions": []
    }
  ]
}
```

### 11.4 互換性方針
- 設定ファイルは `SchemaVersion` を必須とする。
- 未知のスキーマバージョンは安全に読み込み失敗とし、ユーザにエラーを通知する。
- 将来バージョンで移行処理を追加できる構造にする。

## 12. アーキテクチャ設計
### 12.1 レイヤ構成
- WPF UI
- Tray Host
- Application Services
- Domain
- Infrastructure

### 12.2 採用パターン
- UI は MVVM を採用する。
- DI とアプリライフサイクル管理には `.NET Generic Host` を採用する。
- ドメインルールは UI と Infrastructure から分離する。

### 12.3 主要コンポーネント
#### Domain
- `ProfileDefinition`
- `ActionDefinition`
- `CheckDefinition`
- `PostCheckDefinition`
- `SwitchPlan`
- `SwitchExecutionResult`
- `AppState`

#### Application Services
- `ISwitchOrchestrator`
- `IActionRunner`
- `ICheckEvaluator`
- `IWallpaperService`
- `IExecutionLogService`
- `IProfileRepository`
- `IAutoStartService`

#### Infrastructure
- JSON 設定永続化
- Windows サービス操作
- プロセス起動
- 壁紙変更 API 呼び出し
- Windows タスクスケジューラ操作
- ローカルファイルログ出力

### 12.4 コンポーネント責務
#### `ISwitchOrchestrator`
- `SwitchAsync(targetProfileId)` を公開する。
- 切替全体確認を要求する。
- アクション列を順次実行する。
- 失敗時のユーザ選択に従って再試行、続行、中止を制御する。
- 状態遷移とログ記録を統括する。

#### `IActionRunner`
- アクション種別ごとの実行を抽象化する。
- 実行結果として `Succeeded` `Failed` `Skipped` `TimedOut` `Cancelled` を返す。

#### `ICheckEvaluator`
- 事前条件チェックと `CommandCheck` 型の事後確認を評価する。

#### `IWallpaperService`
- カスタム画像適用
- プロファイル名からの画像生成
- キャッシュ管理

#### `IProfileRepository`
- 設定の読込・保存を担う。
- スキーマバージョン整合性を検証する。

#### `IAutoStartService`
- `StartWithWindows` の有効/無効を Windows タスクスケジューラへ反映する。
- 現在の登録状態を取得できる。

### 12.5 実行シーケンス
1. `ISwitchOrchestrator` が切替要求を受ける。
2. 実行中フラグを確認し、処理中なら拒否する。
3. 切替確認ダイアログを表示する。
4. 現在プロファイルの終了時アクションを順次実行する。
5. 切替先プロファイルの起動時アクションを順次実行する。
6. `IWallpaperService` で壁紙を更新する。
7. `AppState` と `LastActiveProfileId` を更新する。
8. `IExecutionLogService` が結果を保存する。

## 13. 実装上の設計判断
### 13.1 権限
- アプリ本体は管理者権限で常駐する。
- 将来互換のためアクション単位で `RunLevel` を保持する。
- v1 では `Standard` / `Elevated` のプロセス起動分離は実装しない。

### 13.2 切替競合
- 切替要求のキューイングは行わない。
- 実行中に別切替要求が来た場合はエラーメッセージを表示し、受け付けない。

### 13.3 ログ保持
- `LogRetentionDays` に基づき古いログを削除できるようにする。
- v1 では起動時またはログ出力時の簡易クリーンアップでよい。

### 13.4 自動起動
- `StartWithWindows` は Windows タスクスケジューラで実装する。
- ログオン時に最高権限で起動する 1 ユーザ向けタスクを登録対象とする。

## 14. 受け入れ基準
- ユーザはタスクトレイからプロファイルを選択し、確認ダイアログ経由で切替できる。
- 終了時アクションと起動時アクションは定義順で実行される。
- `RunLevel` を設定として保持できる。
- 事前条件チェックにより不要アクションをスキップできる。
- 事後確認により期待状態の到達確認ができる。
- 切替結果と失敗理由がログで追跡できる。
- 壁紙がプロファイルごとに切り替わる。
- 起動時は前回プロファイルを表示するが自動再実行しない。
- 切替中の再切替要求は拒否される。
- `StartWithWindows` によりログオン時自動起動を有効/無効にできる。

## 15. テスト計画
### 15.1 単体テスト
- プロファイル設定の JSON 読込・保存
- `SchemaVersion` 判定
- 事前条件チェック判定
- 事後確認のポーリング判定
- 状態遷移
- 切替順序制御

### 15.2 結合テスト
- タスクトレイからの切替操作
- 終了時アクションから起動時アクションへの順次実行
- 個別確認ダイアログ分岐
- 失敗時の `再試行` `残りを続行` `中止`
- 壁紙更新
- ログ出力

### 15.3 シナリオテスト
- プロファイル CRUD が保存後も維持されること
- `ApplicationLaunch` が対象プロセスを起動できること
- `CommandExecution` が出力と終了コードを取得できること
- `WindowsServiceControl` がサービスの開始/停止を制御できること
- 事前条件チェック失敗時に `Skipped` で記録されること
- 事後確認失敗時に切替が停止し、ユーザ選択に応じて分岐すること
- 壁紙未指定時に自動生成画像が作成されること
- 再起動後に前回プロファイル名が復元されること
